# Story Generation & Evolution — Implementation Plan

This plan describes how the game evolves the story across groups using **eager, early-triggered generation**. It builds on `GROUP_ARCHITECTURE.md` (which supersedes the linear 5-scene model in `BRIEF.md`).

---

## Core idea

Story is delivered in **3 groups of ~4 scenes**. Each group is generated by an LLM call that takes the player's prior choices as context. Generation for Group N+1 fires **at the start of Scene 3 of Group N** — early enough that by the time the player reaches the twist card, Group N+1 and all its assets are fully ready.

The twist card is therefore a **purely narrative beat**, never a loading screen.

```
Group 1 (generated upfront)
  Scene 1 → Scene 2 → Scene 3 ──┐
                                │ fire /api/generate-group for Group 2
                                │   (with choice history of Scenes 1–2)
                                │
                  Scene 4       │ Group 2 + assets generating in background
                                │
              Twist Card ───────┘ (Group 2 ready by now)

Group 2
  Scene 5 → Scene 6 → Scene 7 ──┐
                                │ fire /api/generate-group for Group 3
                  Scene 8       │
              Twist Card ───────┘

Group 3
  Scene 9 → ... → Scene 12
                                ──→ fire /api/generate-epilogue
              Ending + Share Card
```

---

## Trade-off (locked)

Group N+1 is shaped by choices from **Scenes 1–2 of Group N only** — the last two choices in the group don't influence the next group's content. This is acceptable because:

- Two choices give enough signal to pick a tonal direction (hype-leaning vs integrity-leaning) and which archetypes to bring forward.
- The twist card text can reference the most recent choice as a post-hoc narrative patch.
- The alternative (waiting until the last choice) forces the twist card back into being a loading buffer.

---

## LLM calls (4 total)

| Call | When | Inputs | Output |
|------|------|--------|--------|
| `generate-group` (Group 1) | End of intro phase | startup, persona, flavorTags | Group JSON: twistCard, scenes[] |
| `generate-group` (Group 2) | Scene 3 of Group 1 mounts | + choices from Scenes 1–2, stats | same shape |
| `generate-group` (Group 3) | Scene 3 of Group 2 mounts | + choices from Group 1 + Scenes 5–6, stats | same shape |
| `generate-epilogue` | After ending classifier fires | full choice history, ending key, flavor tags | ~80 tokens of named-choice prose |

Model: **Anthropic Sonnet 4.6**. Validated server-side with Zod. Stat deltas clamped to ±2 regardless of LLM output. One retry on parse failure, then fallback.

## Implementation order

### 1. Extend types (`src/lib/types.ts`)
- Add `Group { twistCard: string, scenes: Scene[], status: 'pending' | 'ready' | 'failed' }`
- Replace `StoryArc.scenes: Scene[]` with `StoryArc.groups: Group[]`
- Keep `Scene` shape as-is.

### 2. Extend session state (`src/lib/session.ts`)
- Add phase: `'twist-card'` between `'scene'` and `'ending'`.
- Add `groupIndex: number` to `SceneProgress`.
- Add actions: `groupReady(groupIndex, group)`, `enterTwistCard()`, `exitTwistCard()`.
- `chooseOption` and `advanceScene` index into `arc.groups[groupIndex].scenes[sceneIndex]`.

### 3. Build `/api/generate-group`
- Route file: `src/app/api/generate-group/route.ts`.
- Body: `{ groupIndex, startupName, founderPersona, flavorTags, priorChoices, currentStats }`.
- Anthropic SDK with prefilled `{` to force JSON.
- Zod schema validates: scenes array length 3–5, choice labels ≤8 words, deltas ∈ {−2,−1,+1,+2}.
- Server-side clamp on deltas as a safety net regardless of validation.
- Retry once on parse failure. On second failure, return fallback group from `src/lib/fallback-groups/group-{N}.json`.

### 4. Build `/api/generate-epilogue`
- Route file: `src/app/api/generate-epilogue/route.ts`.
- Body: `{ choiceHistory, endingKey, flavorTags, startupName }`.
- ~80 tokens, names specific choices and SF cameos encountered.
- Persist to `playthroughs.epilogue` (add column in a new migration).

### 5. Authored fallback groups
- One static JSON file per group slot in `src/lib/fallback-groups/`.
- The current hardcoded Wagr scenes from `src/app/page.tsx:47-183` become Fallback Group 1 (split into 4 scenes).
- Used silently if generation fails or times out (>10s).

### 6. Eager trigger wiring (`src/app/page.tsx`)
- When `sceneIndex` within a group becomes 2 (Scene 3 mounts) AND `groupIndex < 2` AND `arc.groups[groupIndex+1].status !== 'ready'`:
  - Fire `/api/generate-group` for `groupIndex + 1`.
  - On response: call `groupReady`, then fan out asset generation.
- Edge case: on session rehydrate mid-Scene 3, do not refire if Group N+1 already exists in state.

### 7. Parallel asset fan-out
- As soon as a group's JSON arrives:
  - `Promise.all` over `imagePrompt`s → existing `/api/generate-image` batch endpoint.
  - `Promise.all` over dialogue lines for TTS (ElevenLabs).
- Write `imageUrl` and `audioUrl` back into the scene objects in state as each resolves.
- Failure of an individual asset does not block playback — scene falls back to text-only or no portrait.

### 8. Twist card handling
- Phase transitions to `'twist-card'` after the last scene of a group finishes.
- Render `arc.groups[groupIndex].twistCard` text + ambient animation.
- Hold for **minimum 3s**.
- Exit condition: `groupIndex+1` is `'ready'` AND minimum elapsed.
- Failsafe: if still `'pending'` after 8s, swap in fallback group silently and proceed.

### 9. Group 1 generation timing
- Generate Group 1 during the existing `'generating'` phase (between intro and first scene).
- This is the only loading screen the player ever sees. Cinematic copy already covers it.

### 10. Persistence
- Save full `arc` JSON to `playthroughs.arc_json` on first group ready.
- Patch `arc_json` after each subsequent group lands.
- Save `epilogue` to a new `playthroughs.epilogue` column (migration `0002_add_epilogue.sql`).

---

## Open items to confirm before coding

1. **Group sizes.** Plan assumes 4 scenes per group. Variable lengths (3–5) are allowed by `GROUP_ARCHITECTURE.md`; pick fixed-4 for v1 to simplify trigger logic.
2. **Scene 3 free-text counter-offer.** UI exists in `TextInputPanel` but unwired. For v1, treat counter-offer text as the "walk" choice (binary fallback). Defer `/api/classify-counter-offer` to v2.
3. **Achievements.** Triggered by behavior in playthrough (timeouts, stat thresholds, cameo encounters). Compute client-side from `history` + `stats` at ending. Out of scope for the generation pipeline itself.

## First slice to ship

1. Types + session state changes (steps 1–2).
2. `/api/generate-group` with Zod + fallback (step 3).
3. Wire Group 1 generation in the existing `'generating'` phase (step 9).

This gets the LLM into the loop end-to-end with one group. Eager trigger, twist card, and Groups 2–3 layer on after that works.
